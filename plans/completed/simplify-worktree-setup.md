# Simplify Worktree Setup

## Problem

Setting up a new worktree requires 6 manual steps (documented in `plans/worktree-setup.md`), several of which are workarounds for fixable root causes. Three of the six steps shouldn't be necessary at all.

## Root Cause Analysis

### Step 4 (esbuild install.js hack) — misconfigured `onlyBuiltDependencies`

The `onlyBuiltDependencies` setting lives in `core/sdk/pnpm-workspace.yaml`, which is the wrong file. pnpm reads this from `package.json` under the `"pnpm"` key, not from `pnpm-workspace.yaml`. Because pnpm never actually reads it, esbuild's postinstall is blocked by the default `approve-builds` gate, which requires interactive TTY input that can't be scripted.

**Fix:** Move `onlyBuiltDependencies` into each subpackage's `package.json` that uses esbuild (agents, core/sdk, server), and delete the bogus `core/sdk/pnpm-workspace.yaml`. Then `pnpm install` will run esbuild's postinstall automatically — no manual `install.js` needed.

### Step 5 (bootstrap `dist/index.d.ts`) — hand-maintained ghost artifact

`core/sdk/dist/index.d.ts` is not tracked by git (`.gitignore` excludes `dist/`) and is not generated by any build step. It's a manually maintained stripped copy of `core/sdk/types.ts`. The build script blindly does `cp dist/index.d.ts ../../sdk-types.d.ts`, which fails on a fresh worktree because the file doesn't exist.

**Fix:** Change the sdk build command to copy from the source file `types.ts` instead of from the phantom `dist/index.d.ts`. The `types.ts` file IS tracked in git and always present. The build becomes:

```
esbuild runner.ts --bundle ... && cp types.ts ../../sdk-types.d.ts
```

Then delete the orphaned `dist/index.d.ts` since nothing generates it.

> `types.ts` has JSDoc comments while `dist/index.d.ts` strips them — but the consumer (Tauri app bundling for user-facing SDK types) benefits from having the docs, so this is actually better.

### Step 6 (verify all builds) — unnecessary for worktree setup

The `dev` command already runs `build:sdk`, `build:agents`, and `build:migrations` (either as watch tasks in HMR mode, or as a sequential pre-build in no-HMR mode). Running them manually to "verify" before running `pnpm dev` is redundant. Remove this step from the setup guide.

### Steps 1-3 — genuinely needed, should be scripted

1. **Copy** `.env` — env files aren't in git, must come from somewhere
2. **Root** `pnpm install` — always needed
3. **Subpackage** `pnpm install` — needed because there's no root pnpm workspace

Step 3 exists because each subpackage manages its own `node_modules` independently. A pnpm workspace *could* eliminate this, but the subpackages have divergent dependency versions (e.g., `zod@3` in server vs `zod@4` elsewhere, `ai@5` in root vs `ai@6` in agents) making workspace migration non-trivial and out of scope here.

## Phase 1: Fix esbuild `onlyBuiltDependencies`

In each subpackage that depends on esbuild, add to `package.json`:

`agents/package.json` — uses esbuild via tsup:

```json
"pnpm": {
  "onlyBuiltDependencies": ["esbuild", "@esbuild/darwin-arm64"]
}
```

`core/sdk/package.json` — uses esbuild directly:

```json
"pnpm": {
  "onlyBuiltDependencies": ["esbuild", "@esbuild/darwin-arm64"]
}
```

`server/package.json` — uses esbuild (check if it actually does — server build is `tsc`, may not need esbuild at all):

- Server's build script is just `tsc`. Check if esbuild is even a dependency. If not, skip.
- If it's a transitive dep (via tsx), it may still need the config.

Delete `core/sdk/pnpm-workspace.yaml` (it only contains the misplaced config).

**Verify:** After this change, `cd agents && rm -rf node_modules && pnpm install` should install esbuild platform binaries without needing manual `install.js`.

## Phase 2: Fix sdk build artifact

In `core/sdk/package.json`, change:

```json
"build": "esbuild runner.ts --bundle --format=esm --target=node22 --platform=node --outfile=../../sdk-runner.mjs --sourcemap && cp dist/index.d.ts ../../sdk-types.d.ts"
```

To:

```json
"build": "esbuild runner.ts --bundle --format=esm --target=node22 --platform=node --outfile=../../sdk-runner.mjs --sourcemap && cp types.ts ../../sdk-types.d.ts"
```

Delete `core/sdk/dist/index.d.ts` (orphaned, untracked).

**Verify:** `pnpm build:sdk` succeeds in a clean state without needing `dist/` to pre-exist.

## Phase 3: Create setup script

`scripts/setup-worktree.sh`:

```bash
#!/bin/bash
set -e

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
WORKTREE_ROOT="$(dirname "$SCRIPT_DIR")"
MAIN_WORKTREE="${1:-$HOME/Documents/juice/anvil/anvil}"

echo "Setting up worktree at: $WORKTREE_ROOT"
echo "Copying .env from: $MAIN_WORKTREE"

# 1. Copy .env
if [ ! -f "$MAIN_WORKTREE/.env" ]; then
  echo "ERROR: No .env found at $MAIN_WORKTREE/.env"
  echo "Usage: ./scripts/setup-worktree.sh [path-to-main-worktree]"
  exit 1
fi
cp "$MAIN_WORKTREE/.env" "$WORKTREE_ROOT/.env"

# 2. Install all dependencies
cd "$WORKTREE_ROOT"
pnpm install

SUBPACKAGES=(agents core/sdk migrations api server)
for pkg in "${SUBPACKAGES[@]}"; do
  if [ -f "$pkg/package.json" ]; then
    echo "Installing $pkg..."
    (cd "$pkg" && pnpm install)
  fi
done

echo "Setup complete. Run 'pnpm dev' to start."
```

Add to root `package.json`:

```json
"setup": "./scripts/setup-worktree.sh"
```
