Why Build Agent Tooling?

The business case

  • We all spend most of our time writing code
  • If we build tooling that makes us measurably faster, we are better at our jobs
  • Better tooling = competitive advantage.
    The teams that figure out agent-assisted development first win.
  • This isn't theoretical — we're already using it daily to ship Mort itself

Why a Desktop App?

Why not just extend Claude Code?

  • Hooks are powerful — PreToolUse, PostToolUse, custom timeouts, deny overrides
  • But hooks are reactive, not architectural. You can't:
    • Control how agents are spawned, isolated, or coordinated
    • Build custom permission modes with runtime switching
    • Track state across parallel agents and sub-agents
    • Build a persistent plan system with automatic detection
    • Own the analytics pipeline
  • Hooks let you add guardrails. A client lets you build the rails.
  • Full control: macOS-native panels, global hotkeys, system tray, live streaming
  • We track the SDK — adopt new capabilities day-one

Claude Code Permission System

Layers of defense

Command parsing

Bash commands are parsed to detect chaining (&&, ||, ;). Each sub-command is evaluated independently — no sneaking rm -rf after a benign ls.

Evaluation order
  1. Hooks (can deny, even in bypass)
  2. Deny rules (hard block)
  3. Allow rules (auto-approve)
  4. Ask rules (prompt user)
  5. Permission mode
  6. canUseTool (last resort)
Key insight

A deny at any layer overrides all allows below it. Defense in depth.

Writer Contract

Complete disk write before emitting event. Listeners always re-read from disk, never trust event payloads.

Events as Signals, Disk as Truth

~/.mort/
├── agent-hub.sock                # Unix socket (runtime)
├── threads/{id}/
│   ├── metadata.json             # Status, git info, timestamps
│   ├── state.json                # Messages, tools, sessionId (resume)
│   └── draft.txt                 # Unsent user input
├── plans/{id}/metadata.json      # Plan tracking
├── plan-thread-edges/            # {planId}-{threadId}.json
├── repositories/{slug}/          # Worktrees, thread branches
├── settings/config.json          # Hotkeys, preferences
└── databases/
    ├── clipboard.db              # Clipboard history (FTS)
    └── drain-events.db           # Analytics (17 event types)
  • Events are signals, not data — they notify that something changed, never carry the truth
  • Disk is the source of truth — listeners always re-read from disk on signal
  • Multi-writer safe: UI + agents both write
  • Crash resilient: no in-memory-only state
  • Complete disk write before emitting event

Testing Strategy

Three isolation levels

LevelWhat's realWhat's mockedSpeed
E2E (Tauri)Full app + agentsNothing~seconds
HarnessAgent subprocessHub socket~ms
Happy-DOMComponent logicProcesses, DOM~ms

Why tests are faster on a new codebase

  • Fewer people = fewer conflicts — reshape test infra without coordinating
  • Agents have log access by default — logs/dev.log always available
  • Agent-written tests — agents write, run, and iterate in a closed loop
  • Clean boundaries — adapter pattern = every boundary is injectable/mockable

Each level tests a different trust boundary. E2E proves the system works. Harness proves agent behavior. Happy-DOM proves component logic.

Code Quality: Static Analysis

No execution required

  • Lines of code
  • Cyclomatic complexity
  • Linters (ESLint, Clippy)
  • Type coverage
  • Dependency analysis
  • Dead code detection
  • Code churn (frequency of changes to a file)
  • Halstead volume
  • Microsoft Maintainability Index

Cheapest to run. Every commit. No excuses.

Cyclomatic complexity — the number of linearly independent paths through a function. Each if, for, while, case, &&, || adds a path. CC of 1 = no branching. CC > 10 = hard to test and reason about.
Halstead volume — measures program size through operators and operands. V = N × log2(n) where N = total operators + operands, n = unique operators + operands. Higher volume = more cognitive load.

Code Quality: Dynamic Analysis

Requires running the code

  • Test coverage (statement, branch, path)
  • Integration test results
  • Performance benchmarks
  • Memory profiling
  • Fuzzing
  • Load testing
  • E2E tests
  • Defect rate (bugs per KLOC or per commit)

Proves things work. More expensive, but this is where confidence comes from.

Production observation

  • Error rates
  • Performance metrics (p50, p95, p99)

The Equation of Software Design

Max Kanat-Alexander — codesimplicity.com

D = (Pv × Vi) / (Ei + Em)
VariableMeaning
DDesirability of implementation
PvProbability of value — will it actually be useful?
ViPotential value — how valuable could it be?
EiEffort of implementation — one-time build cost
EmEffort of maintenance — ongoing cost over time
The key insight

As time approaches infinity, Em dominates. Implementation effort is paid once. Maintenance effort is paid forever.

This is why simplicity matters — it determines the coefficient of maintenance.

With AI agents, Ei drops dramatically. But Em stays the same — or gets worse. The equation says: when implementation is cheap, maintenance is all that matters.

Why this matters more with agents

  • Agents dramatically reduce implementation effort
  • Maintenance effort doesn't shrink at the same rate — and can grow if we're not deliberate
  • Agent code can trend toward: more abstractions than needed, inconsistent patterns across sessions, code that grows faster than it gets pruned
  • State compounds the problem — it's not a closed system of code, it's code + state. Each new feature multiplies the state space. Agents add features fast, which means state explosion becomes a real risk.
  • When implementation is cheap, maintenance dominates. YAGNI matters more with AI, not less.

Composite Metrics

Combining signals for better risk detection

CRAP = CC2 · (1 - coverage)3 + CC

CRAP metric — penalizes complex untested code. High cyclomatic complexity + low test coverage = high CRAP score.

  • CC = 1, coverage = 100% → CRAP = 1 (trivial, tested)
  • CC = 30, coverage = 0% → CRAP = 930 (complex, untested)
  • Useful as a quality gate: reject code above a CRAP threshold
Risk = Complexity × Churn

Hotspot analysis (Adam Tornhill) — files that are both complex and frequently changed are the highest-risk targets in a codebase.

  • Complex + rarely changed = stable (low risk)
  • Simple + frequently changed = fine (low risk)
  • Complex + frequently changed = hotspot (refactor here)
  • Agents can compute this from git log + static analysis
Both metrics are agent-computable — no human judgment required. Perfect for automated quality gates in the agent loop.

LLM-Bound Quality Techniques

Using AI to judge AI

LLM as reviewer
  • Code review: single responsibility, naming, edge cases
  • Architecture review: pattern consistency
  • Intent verification: does implementation match plan?
LLM as quality gate
  • Pre-merge review agent in Approve mode
  • Diff summarization to reduce review burden
  • Regression detection: modified behavior → updated tests?
Agentic quality techniques
  • Agent writes → different agent reviews (adversarial)
  • Agent writes tests → run → iterate (closed loop)
  • Agent refactors → test suite → revert if regression
  • Plan mode → approve plan → implement → review

Formal Methods

Mathematical proof that software is correct

  • Formal verification — prove properties about code mathematically, not just test them
  • Proof assistants (Coq, Lean, Isabelle) — write machine-checked proofs that your code satisfies a specification
  • Model checking — exhaustively verify all possible states of a system
  • Historically too expensive for most software — reserved for aerospace, crypto, kernels
Why this matters now
  • LLMs can generate formal proofs — dramatically reducing the human effort barrier
  • Agent writes code → agent writes proof → proof assistant verifies
  • If it compiles in Coq, it's mathematically guaranteed correct
  • Moves us from "tests pass" to "provably correct"

What Makes Code High Quality?

Quality = human-readable AND machine-readable.
Quality = correct today AND changeable tomorrow.
Quality = minimum complexity for current requirements (YAGNI).
The best code is code that both humans and agents can reason about confidently.
  • Small files, clear naming, explicit types, minimal magic
  • Code that ships, doesn't break, and is easy to change
  • New metric worth thinking about: AI-navigability — how quickly can an agent locate, understand, and safely modify this code?
  • DORA metrics (deploy frequency, change failure rate, time to recovery) matter more than code metrics alone
